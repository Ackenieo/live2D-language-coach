# AI 视觉对话助手提交设计文档

## 1. 项目定位

本项目实现的是 `TalkTic / Live2D Language Coach`：一个面向英语口语练习的 AI 视觉对话助手。用户在浏览器中打开麦克风和摄像头，与 AI 进行实时语音对话；AI 可以结合语音和摄像头截图作答，前端用 Live2D 角色承载语音、字幕、嘴型和表情反馈，后端负责实时对话、用户体系、会话报告、评分和排行榜。

## 2. 用户故事计划与最终实现

| 用户故事 | 计划实现 | 最终实现情况 |
|---|---|---|
| 用户可以用手机号验证码登录，首次登录自动创建账号 | 是 | 已实现。后端提供 `/api/auth/sms/send`、`/api/auth/login`、`/api/auth/refresh`，登录后签发 access token 和 refresh token。 |
| 用户可以维护个人资料 | 是 | 已实现。支持查询资料、修改昵称、上传头像到 OSS、换绑手机号。 |
| 用户可以进入实时通话页并打开麦克风 | 是 | 已实现。前端采集 16 kHz PCM Int16 音频，通过 `/ws/bailian` WebSocket 发送给后端。 |
| AI 可以听到用户说话并实时回复语音 | 是 | 已实现。后端转发音频到百炼 Realtime API，并把 AI 返回的 24 kHz PCM 音频通过 WebSocket 二进制帧推给前端播放。 |
| AI 回复时可以显示实时字幕 | 是 | 已实现。后端转发 `ai_subtitle`、`ai_subtitle_complete`、`user_subtitle` 等事件，前端同步展示字幕。 |
| 用户可以打开摄像头，让 AI 获取视觉上下文 | 是 | 已实现基础版。前端发送低清 JPEG 截图，后端在视觉模式开启时把最新截图作为图片上下文转发给模型。 |
| AI 可以持续理解完整视频流 | 评估后决定 | 未采用连续视频流。最终采用“截图抽帧 + 去重 + 每轮只取最新帧”，满足题目视觉要求并控制成本。 |
| Live2D 角色可以跟随 AI 语音产生嘴型和表情 | 是 | 已实现。前端根据实际播放音频能量和字幕元音生成嘴型，根据 AI 文本推断表情。 |
| 用户可以选择练习场景、难度和口音 | 是 | 部分实现。后端提示词模板和 WebSocket 配置已支持场景、难度、口音；前端当前以默认配置为主。 |
| 每轮对话后用户可以看到发音评分 | 是 | 已实现。后端接入腾讯智聆并发布 `pronunciation_score` 事件，前端展示综合、准确度、流利度、完整度等级。 |
| 用户可以看到语法纠错反馈 | 是 | 已实现。后端调用文本模型做语法纠错，并通过 `grammar_correction` 事件返回给前端。 |
| 用户结束会话后可以得到总结报告 | 是 | 已实现。`finish` 后聚合会话时长、消息数、评分和建议，生成 `session_end` 并落库，前端进入总结页。 |
| 用户可以查看历史会话 | 是 | 已实现。提供 `/api/chat/history` 和 `/api/chat/report/{sessionId}`。 |
| 用户可以查看排行榜和自己的排名 | 是 | 已实现。提供 `/api/leaderboard` 和 `/api/leaderboard/my-rank`。 |
| 用户断线后可以恢复同一会话 | 是 | 后端已支持 `reconnectSessionId` 和 `reconnect`，前端当前未把它做成主要可见入口。 |
| 系统可以限制高频调用，避免接口被滥用 | 是 | 已实现基础限流。Sentinel 配置覆盖短信、登录、用户资料、报告、排行榜和 WebSocket 资源。 |

## 3. 控制运营成本的思路

### 3.1 想到过的技巧

| 成本来源 | 控制技巧 |
|---|---|
| 摄像头画面上传 | 降低分辨率、降低 JPEG 质量、抽帧、相似帧去重、摄像头开关默认关闭、每轮只取最新帧。 |
| 多模态模型上下文 | 只在用户开启视觉模式后发送图片；只在用户轮次发送视觉上下文；提示词约束 AI 不要无关描述画面。 |
| 语音传输 | 使用 WebSocket 流式传输；音频走二进制帧，避免 base64 放大体积。 |
| AI 回复延迟 | 音频、字幕、评分、报告分离；音频优先返回，评分和报告异步到达。 |
| 嘴型同步 | 不要求云端额外生成 lip-sync JSON，前端本地根据播放音频和字幕估算嘴型。 |
| 会话记忆 | Redis 存短期记忆，配置 TTL 和最大消息数，结束或清理会话时释放。 |
| 发音评分和语法纠错 | 只对有效用户轮次触发，放到异步事件链路，避免阻塞主对话链路。 |
| 短信和登录滥用 | 对短信、登录、刷新 token 等接口做限流，减少恶意调用成本。 |
| WebSocket 并发 | 对 `/ws/bailian` 握手和资源名做限流，避免实时模型连接被打爆。 |
| 静态资源 | Live2D 模型放在前端静态目录，由浏览器缓存；不通过后端反复转发。 |
| 外部服务配置 | API key、OSS、评测服务等配置化，缺少配置时可以降级或跳过非核心能力。 |

### 3.2 实际采用的技巧

- 摄像头截图压缩为 `240x180`，JPEG 质量为 `0.2`，避免上传高分辨率图像。
- 前端用 pHash 做相似帧去重，相似画面不重复发送。
- 摄像头由独立开关控制，关闭后系统只走语音对话链路。
- 后端 `FrontendImageQueue` 只保留有限数量图片，并在用户轮次中取最新帧给模型。
- AI 音频从后端到前端使用 WebSocket `BinaryMessage`，没有转成 JSON/base64。
- 嘴型同步在浏览器端完成，没有增加额外云端 lip-sync 调用。
- Redis 会话记忆配置了最大消息数和 TTL，避免长期堆积上下文。
- 发音测评和语法纠错通过异步事件返回，不阻塞实时语音回复。
- Sentinel 已配置 REST 和 WebSocket 资源限流，覆盖短信、登录、用户资料、报告、排行榜、`WS:/ws/bailian` 等接口。
- 会话结束后再聚合报告和写库，实时链路只保留必要状态，减少主链路压力。

## 4. 未采用但后续可补充的成本优化

- 增加端侧轻量视觉检测，只在画面语义明显变化时上传截图。
- 根据网络质量、模型负载和用户套餐动态调整截图频率。
- 增加用户级配额，例如每日会话时长、每日视觉帧数量、每日评分次数。
- 增加运营成本看板，统计每个用户、场景和接口的模型调用量。
- 使用 Service Worker 预缓存 Live2D 资源，降低弱网下的重复加载成本。
